## 1. 실시간 통신: WebSocket vs SSE

### 🔍 문제 상황

골프 장비 P2P 렌탈 플랫폼에서 필요한 실시간 기능:
- **채팅**: 판매자-구매자 간 거래 협상을 위한 양방향 실시간 소통
- **알림**: 예약 요청, 결제 완료, 새 메시지 도착 등을 즉시 전달

초기에는 WebSocket만으로 모든 실시간 통신을 처리하려 했으나, 알림은 서버→클라이언트 단방향만 필요해서 과도한 스펙이었습니다. 사용자 1,000명 기준 불필요한 양방향 연결 유지는 서버 리소스 낭비로 이어졌습니다.

### 📊 기술 검토

| 비교 항목      | WebSocket         | SSE                   | 선택           |
|------------|-------------------|-----------------------|--------------|
| **통신 방향**  | 양방향 (Full-Duplex) | 단방향 (Server → Client) | 용도별 선택       |
| **사용 사례**  | 실시간 채팅            | 실시간 알림                | 채팅/알림 분리     |
| **연결 유지**  | 지속적 연결            | 자동 재연결                | SSE 유리       |
| **구현 복잡도** | 높음 (STOMP 등)     | 낮음 (HTTP 기반)          | SSE 유리       |
| **메모리 사용** | 높음                | 낮음                    | SSE 유리       |
| **브라우저 지원** | 모든 브라우저           | 대부분 지원 (IE 제외)        | 문제 없음        |

### 🤔 고민한 점

1. **WebSocket만 사용**: 구현은 단순하지만 알림에 양방향 연결이 불필요 → 리소스 낭비
2. **SSE만 사용**: 채팅 구현 불가능 (클라이언트→서버 통신 필요)
3. **둘 다 사용**: 복잡도 증가하지만 각 용도에 최적화된 기술 적용 가능

### ⚖️ 트레이드오프

**포기한 것:**
- 단일 기술로 통일하는 단순함
- 프론트엔드에서 2개의 연결 관리

**얻은 것:**
- 각 기능에 맞는 최적의 성능과 리소스 효율
- 알림 연결의 자동 재연결 기능 (SSE 기본 제공)
- 메모리 사용량 30% 절감 (SSE가 WebSocket보다 경량)

### ✅ 최종 결정

- **채팅**: WebSocket (STOMP 프로토콜)
  - 양방향 메시지 교환 필요
  - Redis Pub/Sub으로 멀티 서버 동기화

- **알림**: SSE (Server-Sent Events)
  - 서버→클라이언트 단방향 푸시
  - 하트비트 메커니즘으로 좀비 연결 제거

### 📈 결과

| 지표              | 측정값        | 비고                     |
|-----------------|------------|------------------------|
| **채팅 응답 시간**    | <100ms     | Redis Pub/Sub 기반       |
| **SSE 동시 연결**   | 1,000+ 안정  | 하트비트 30초 주기           |
| **메모리 사용량**     | 30% 절감     | WebSocket 대비 SSE 경량화   |
| **연결 재연결 성공률** | 99.9%      | SSE 자동 재연결 기능          |

## 2. AI 챗봇: LangChain4j + Google Gemini

### 🔍 문제 상황

골프 장비 렌탈 플랫폼 사용자들이 자주 묻는 질문:
- "타이틀리스트 드라이버 렌탈 있나요?"
- "예약 취소는 어떻게 하나요?"
- "보증금은 언제 돌려받나요?"

고객센터 FAQ만으로는 한계가 있었고, 게시글 검색도 키워드 매칭이라 정확도가 낮았습니다. **AI 챗봇**으로 자연어 질의응답과 **시맨틱 검색**을 구현하기로 결정했습니다.

### 📊 기술 검토

| 비교 항목          | LangChain4j + Gemini | LangChain + OpenAI | Spring AI + GPT-4 |
|----------------|----------------------|--------------------|-------------------|
| **가격**         | 무료 (Gemini Free Tier) | 유료 ($0.03/1K tokens) | 유료                |
| **응답 속도**      | 빠름 (Flash 모델)        | 보통                 | 보통                |
| **한국어 지원**     | 우수                   | 우수                 | 우수                |
| **Java 통합**    | 네이티브 (LangChain4j)   | Python SDK 의존      | Spring 네이티브        |
| **RAG 구현**     | 쉬움 (기본 제공)           | 쉬움                 | 보통                |
| **Tool Calling** | ✅ 지원                 | ✅ 지원               | ✅ 지원              |

### 🤔 고민한 점

1. **OpenAI GPT-4**: 성능은 검증되었지만 **유료**이고, Python 중심 생태계라 Java 통합이 번거로움
2. **LangChain (Python)**: 생태계가 크지만 **Spring Boot와 별도 서버** 필요 (MSA 복잡도 증가)
3. **Spring AI**: Spring 네이티브지만 **초기 버전**이라 문서와 예제 부족
4. **LangChain4j + Gemini**: Java 네이티브, 무료 tier, 빠른 응답 속도

### ⚖️ 트레이드오프

**포기한 것:**
- OpenAI GPT-4의 검증된 성능과 방대한 생태계
- Spring AI의 Spring 생태계 깊은 통합

**얻은 것:**
- **비용 절감**: 무료 tier로 초기 운영 (월 $0 → $0)
- **빠른 응답**: Gemini Flash 모델 평균 1초 이내
- **네이티브 통합**: Java 코드로 완결, Python 서버 불필요
- **RAG 기본 제공**: Vector Store, Embedding, Tool 추상화

### ✅ 최종 결정

**LangChain4j + Google Gemini 2.0 Flash**

**구현 내용:**
- **Embedding Model**: All-MiniLM-L6-v2 (384차원, 로컬 실행)
- **Vector Store**: Redis Vector Store (RediSearch)
- **LLM**: Google Gemini 2.0 Flash (무료, 빠른 응답)
- **Chat Memory**: Redis 기반 대화 맥락 유지 (최대 10개 메시지)

**Tool Calling 3개:**
1. `searchPost`: 골프 장비 게시글 검색 (벡터 유사도)
2. `searchPolicy`: FAQ/약관 검색 (벡터 유사도)
3. `getPriceStatistics`: 카테고리별 가격 통계

## 3. 벡터 DB: Redis Vector Store

### 🔍 문제 상황

AI 챗봇에서 사용자 질문에 대한 **시맨틱 검색** (의미 기반 검색)이 필요했습니다:
- "타이틀리스트 드라이버 있나요?" → "TITLEIST TSR3 드라이버" 매칭
- "예약 취소 방법" → FAQ "예약 취소 및 환불 정책" 매칭

키워드 기반 검색(LIKE 검색)으로는 **동의어, 오타, 다른 표현**을 처리할 수 없었습니다. 벡터 임베딩으로 변환 후 코사인 유사도 검색이 필수였습니다.

### 📊 기술 검토

| 비교 항목         | Redis Vector Store | Pinecone       | Milvus         | Weaviate |
|---------------|--------------------|----------------|----------------|----------|
| **추가 비용**     | 없음 (기존 Redis 활용)   | $70/월 (Starter) | 무료 (셀프 호스팅)    | 무료       |
| **응답 속도**     | 매우 빠름 (<1ms)       | 빠름 (50-100ms)  | 보통 (100-200ms) | 보통       |
| **운영 복잡도**    | 낮음                 | 낮음 (SaaS)      | 높음             | 중간       |
| **기존 인프라 활용** | ✅ Redis 재사용       | ❌ 외부 SaaS     | ❌ 별도 설치       | ❌ 별도 설치 |
| **데이터 규모**    | ~1M (충분)          | ~10M           | ~100M          | ~10M     |

### 🤔 고민한 점

1. **Pinecone**: SaaS로 운영 부담 없지만 **월 $70 비용** + 외부 의존성
2. **Milvus**: 무료지만 **별도 서버 필요** (Docker 추가) + 운영 복잡도 증가
3. **Weaviate**: 기능은 좋지만 **학습 곡선** + 팀원 러닝 커브
4. **Redis Vector Store**: 이미 캐시/세션용 Redis 사용 중 → **추가 비용 0원**

### ⚖️ 트레이드오프

**포기한 것:**
- Pinecone의 검증된 SaaS 안정성과 대규모 데이터 처리 능력
- Milvus의 고급 벡터 인덱싱 기능 (HNSW, IVF 등)

**얻은 것:**
- **비용 절감**: 별도 벡터 DB 없이 기존 Redis 활용 ($70/월 → $0)
- **운영 단순화**: 추가 인프라 없음, Redis 하나로 통합
- **응답 속도**: 인메모리 기반 <1ms 지연시간
- **팀원 학습 비용 감소**: 이미 사용 중인 Redis 활용

### ✅ 최종 결정

**Redis Vector Store (RediSearch 모듈)**

**구현 내용:**
- **2개의 벡터 인덱스**:
  - `post-embeddings`: 골프 렌탈 게시글 (1,616개)
  - `document-embeddings`: FAQ/약관 문서 (수십 개 청크)
- **임베딩 차원**: 384차원 (All-MiniLM-L6-v2)
- **유사도 메트릭**: Cosine Similarity
- **검색 설정**:
  - 게시글: Top-5, 최소 유사도 0.6
  - 문서: Top-3, 최소 유사도 0.7

### 📈 결과

| 지표            | 측정값                  | 비고                        |
|---------------|----------------------|---------------------------|
| **검색 응답 시간**  | <1ms                 | 인메모리 기반                   |
| **검색 정확도**    | Top-5 평균 유사도 0.85     | 코사인 유사도 기반                |
| **임베딩 크기**    | 1,616개 (Post)         | 384차원 × 1,616 = 약 2.5MB   |
| **메모리 사용량**   | +3MB                 | 기존 Redis에 추가              |
| **비용**        | $0                   | 기존 인프라 활용                 |
| **운영 복잡도**    | 변화 없음                | Redis 하나로 통합              |

## 4. 분산 락: Redisson

### 🔍 문제 상황

멀티 서버 환경(로드밸런서 + EC2 2대)에서 **동시 예약 요청** 발생:

```
시간   | 서버 1 (유저 A)              | 서버 2 (유저 B)
-----|--------------------------|---------------------------
t0   | 12/25 예약 가능 확인 ✅       | 12/25 예약 가능 확인 ✅
t1   | 예약 생성                   | 예약 생성
t2   | DB 저장 성공                | DB 저장 성공
결과   | 💥 같은 날짜에 2개 예약 생성
```

**원인 분석:**
- 애플리케이션 레벨의 중복 체크는 멀티 서버 환경에서 **Race Condition** 발생
- DB 락(SELECT FOR UPDATE)은 성능 저하 + 데드락 위험
- 데이터 일관성 깨짐 → 고객 불만 증가

### 📊 기술 검토

| 비교 항목          | Redisson         | Lettuce (SetNX)    | Database Lock     | Java synchronized |
|----------------|------------------|--------------------|-------------------|-------------------|
| **구현 복잡도**     | 낮음 (추상화 제공)     | 중간 (직접 구현 필요)      | 낮음                | 매우 낮음             |
| **성능**         | 우수               | 우수                 | 보통 (DB 부하)        | 좋음 (단일 서버만)       |
| **Pub/Sub 방식** | ✅ 폴링 없음         | ❌ 폴링 필요           | ❌ 폴링 필요          | ❌ 분산 환경 불가      |
| **자동 해제**      | ✅ Lease Time    | ❌ 수동 구현 필요        | ✅ Timeout         | ✅ finally         |
| **멀티 서버 지원**   | ✅                | ✅                  | ✅                 | ❌                 |
| **CPU 효율성**    | 높음 (이벤트 기반)     | 낮음 (폴링)            | 보통                | 높음                |

### 🤔 고민한 점

1. **Database Lock**: 구현은 쉽지만 **DB 부하 증가** + SELECT FOR UPDATE 성능 이슈
2. **Lettuce SetNX**: 직접 구현 시 **폴링 방식**으로 CPU 낭비 + 코드 복잡도 증가
3. **Java synchronized**: 단일 JVM만 동작 → **멀티 서버 환경에서 무용지물**
4. **Redisson**: Pub/Sub 방식, 자동 해제, 추상화 잘 되어 있음 → 학습 비용만 투자

### ⚖️ 트레이드오프

**포기한 것:**
- Database Lock의 단순함 (ORM만으로 처리 가능)
- Java synchronized의 익숙함

**얻은 것:**
- **성능**: Pub/Sub 방식으로 폴링 없이 즉시 락 획득 (CPU 효율 ↑)
- **안정성**: Lease Time 자동 해제로 데드락 방지
- **멀티 서버 지원**: 분산 환경에서 완벽한 동시성 제어
- **유연한 락 단위**: Post ID 기준 락으로 다른 게시글은 병렬 처리

### ✅ 최종 결정

**Redisson 분산 락**

**구현 내용:**
```java
String lockKey = "reservation:post:" + request.postId();
RLock lock = redissonClient.getLock(lockKey);

try {
    boolean available = lock.tryLock(3, 5, TimeUnit.SECONDS);
    // 대기 시간 3초: 락 획득 시도 최대 대기
    // 자동 해제 5초: 서버 크래시 시 데드락 방지

    if(!available) {
        throw new ReservationException(LOCK_ACQUISITION_FAILED);
    }

    // 원자적 처리 (검증 + 저장)
    validateReservationCreation(...);
    reservationRepository.save(reservation);

} finally {
    if(lock.isHeldByCurrentThread()) {
        lock.unlock();
    }
}
```

**핵심 설계:**
- **Post ID 기준 락**: 같은 장비만 직렬화, 다른 장비는 병렬 처리
- **대기 시간 3초**: 사용자 UX 고려 (3초 이상 대기 시 실패 응답)
- **점유 시간 5초**: 예약 처리 충분 시간 (평균 200-300ms, 여유 20배)

### 📈 결과

| 지표                 | Before      | After      | 개선         |
|--------------------|-------------|------------|------------|
| **중복 예약 발생률**      | 발생 (불특정)    | 0%         | **완전 제거**  |
| **Race Condition** | 발생          | 방지         | **완전 차단**  |
| **멀티 서버 안정성**      | 불안정         | 안정         | **보장**     |
| **성능**             | -           | Post별 독립 락 | **최적화**    |
| **CPU 사용량**        | 폴링으로 높음     | Pub/Sub    | **90% 감소** |

## 5. 메시지 브로커: Redis Pub/Sub

### 🔍 문제 상황

**멀티 서버 환경 (로드밸런서 + EC2 2대)** 에서 실시간 통신 문제:

```
상황: 유저 A가 서버 1에 SSE 연결, 유저 B가 서버 2에서 A에게 채팅 전송

[Server 1]           [Server 2]
유저 A (SSE 연결)       유저 B (메시지 전송)
                       ↓
                    채팅 메시지 전송
                       ↓
                    ❌ 서버 1에 전달 안됨!

결과: 유저 A는 메시지를 받지 못함 💥
```

**원인:**
- 각 서버는 독립된 JVM 프로세스 → **메모리 공유 불가**
- WebSocket/SSE 연결은 서버별로 격리됨
- 서버 간 메시지 브로드캐스트 메커니즘 필요

### 📊 기술 검토

| 비교 항목       | Redis Pub/Sub | RabbitMQ      | Kafka        | AWS SNS/SQS  |
|-------------|---------------|---------------|--------------|--------------|
| **메시지 보장**  | At-most-once  | At-least-once | Exactly-once | At-least-once |
| **응답 속도**   | 매우 빠름 (<1ms)  | 빠름 (5-10ms)   | 보통 (10-50ms) | 보통           |
| **운영 복잡도**  | 낮음            | 중간            | 높음           | 낮음 (SaaS)    |
| **메시지 영속성** | ❌ 없음         | ✅ 디스크 저장     | ✅ 디스크 저장    | ✅            |
| **사용 사례**   | 실시간 알림/채팅    | 작업 큐          | 대용량 스트리밍    | 비동기 작업       |
| **추가 비용**   | $0 (기존 활용)   | $30/월         | $50/월        | 종량제          |

### 🤔 고민한 점

1. **RabbitMQ**: 메시지 영속성 보장하지만 실시간 알림/채팅에는 **오버스펙** + 추가 비용
2. **Kafka**: 대용량 처리에 강하지만 **운영 복잡도 높음** + 실시간성 떨어짐
3. **AWS SNS/SQS**: SaaS로 편하지만 **종량제 비용** + 외부 의존성
4. **Redis Pub/Sub**: 메시지 영속성 없지만 **실시간 통신**에는 충분 + 기존 Redis 활용

**핵심 판단:**
- 채팅/알림 메시지는 **휘발성**이어도 괜찮음 (사용자가 새로고침하면 DB 조회)
- **실시간성**이 가장 중요 (<1ms 지연시간)
- 이미 Redis 사용 중 → **추가 인프라 불필요**

### ⚖️ 트레이드오프

**포기한 것:**
- RabbitMQ/Kafka의 메시지 영속성 보장 (Exactly-once, At-least-once)
- 메시지 재처리 기능

**얻은 것:**
- **초저지연**: <1ms 메시지 전달 (실시간 채팅/알림 최적화)
- **비용 절감**: 기존 Redis 활용 ($30-50/월 → $0)
- **운영 단순화**: 추가 메시지 브로커 불필요
- **멀티 서버 동기화**: Pub/Sub으로 모든 서버에 메시지 브로드캐스트

### ✅ 최종 결정

**Redis Pub/Sub**

**구현 내용:**

**1. 채팅 메시지 브로드캐스트**
```java
// 서버 2: 메시지 전송
chatRedisPublisher.publish("chat:room:123", chatMessage);

// 서버 1, 2 모두: 구독
@RedisMessageListener
public void handleChatMessage(ChatMessage message) {
    // WebSocket으로 클라이언트에게 전달
}
```

**2. 알림 브로드캐스트**
```java
// 서버 1: 알림 생성
notificationRedisPublisher.publish("notification", event);

// 서버 1, 2 모두: 구독
@RedisMessageListener
public void handleNotification(NotificationEvent event) {
    // SSE로 클라이언트에게 전달
}
```

**채널 구조:**
- `chat:room:{roomId}`: 채팅방별 메시지
- `notification`: 전체 알림 이벤트
